PART 3: CENTRAL DATABASE SCHEMA (SUPABASE POSTGRESQL)
================================================================================

The database is built on relational integrity and optimized with indexes to 
prevent double-bookings and speed up queries. Found in `database/schema.sql`:

1. `users` Table:
   - Stores accounts for Students and Admins.
   - Fields: `id` (PK), `name`, `email` (Unique), `password` (bcrypt hash), 
     `role` ('student', 'admin'), and `created_at`.
   - Index: `idx_users_email` (optimizes user authentication checkups).

2. `events` Table:
   - Catalog of events with capacity thresholds and registration limits.
   - Fields: `id` (PK), `title`, `description`, `date`, `venue`, `category`, 
     `price`, `capacity`, `image`, `registration_deadline`, `created_at`.
   - Index: `idx_events_category` (speeds up content recommendations).

3. `registrations` Table:
   - Links users to events with booking statuses.
   - Fields: `id` (PK), `user_id` (FK), `event_id` (FK), `payment_status` 
     ('pending', 'completed', 'failed'), `attendance_status` ('absent', 'present'), 
     `created_at`.
   - Constraint: UNIQUE(`user_id`, `event_id`) prevents double registration.
   - Index: `idx_registrations_user_event`, `idx_registrations_event_id`.

4. `payments` Table:
   - Financial audit logs for Razorpay checkouts.
   - Fields: `id` (PK), `user_id` (FK), `event_id` (FK), `razorpay_order_id`, 
     `razorpay_payment_id`, `razorpay_signature`, `amount`, `status`, `created_at`.
   - Indexes: `idx_payments_user_id`, `idx_payments_event_id`.

5. `tickets` Table:
   - Boarding pass generation reference.
   - Fields: `id` (PK), `registration_id` (FK, Unique), `ticket_number` (Unique), 
     `qr_code` (contains payload/ticket string), `created_at`.
   - Index: `idx_tickets_number`.

6. `attendance` Table:
   - Tracks check-ins scanned at the entrance gate.
   - Fields: `id` (PK), `ticket_id` (FK, Unique), `checkin_time`.
   - Index: `idx_attendance_checkin_time`.

PART 4: BACKEND API ENDPOINTS & LOGIC (routes/ & server.js)
================================================================================

1. Authentication Endpoint (/api/auth) - `backend/routes/auth.js`
   - POST `/register`: Accepts name, email, and password. Hashes passwords 
     using bcryptjs, inserts into `users`, sends a welcome email via Nodemailer.
   - POST `/login`: Verifies user credentials, returns signed JWT.
   - GET `/me`: Returns current user profiles by verifying token.

2. Events Endpoint (/api/events) - `backend/routes/events.js`
   - GET `/`: Fetches events sorted by date. Includes optional query filters.
   - GET `/recommendations`: Content-based filtering matching tags and pricing.
   - GET `/:id`: Fetches detailed metadata of a specific event.
   - POST `/`: (Admin Only) Creates a new event.
   - PUT `/:id`: (Admin Only) Edits event settings.
   - DELETE `/:id`: (Admin Only) Cancels and deletes an event.

3. Registrations Endpoint (/api/registrations) - `backend/routes/registrations.js`
   - POST `/create`: Initiates checkout. Checks deadlines and capacities.
     * Free Events: Registers immediately, generates ticket number, emails PDF.
     * Paid Events: Inserts a pending registration, communicates with Razorpay 
       API to create an Order ID, creates a pending payment row.
   - POST `/verify-payment`: Confirms transactions. Validates the signature, 
     updates registration and payment status, generates ticket number, and dispatches 
     payment/ticket emails.
   - GET `/history`: Retrieves student registration logs, ticket codes, and attendance.
   - DELETE `/:id/cancel`: Deletes a registration if before the deadline.
   - GET `/event/:eventId`: (Admin Only) Fetches rosters for exporting to CSV.

4. Tickets Endpoint (/api/tickets) - `backend/routes/tickets.js`
   - GET `/:id`: Fetches boarding pass metadata. Restricts non-owners.

5. Attendance Endpoint (/api/attendance) - `backend/routes/attendance.js`
   - POST `/scan`: Receives scanned QR codes. Performs validation:
     * Check 1: Does the ticket exist in the database?
     * Check 2: Has it already been scanned? (Avoid double entry check).
     * Check 3: Is registration fully paid?
     * Action: Inserts check-in timestamp and updates status to 'present'.

6. Analytics Endpoint (/api/analytics) - `backend/routes/analytics.js`
   - GET `/dashboard`: (Admin Only) Gathers KPIs and projections:
     * KPI totals: registrations, events, revenue, attendance rate.
     * Historical data: category percentages, daily registrations, monthly sales.
     * Linear Regression Forecast: Computes next 3 months revenue.
     * Peak Heatmap: Groups check-in logs into 5 hour-blocks by weekday (0-6).
     * Popularity Alert: Analyzes booking velocities to predict event sellouts.

PART 5: FRONTEND APPLICATION DETAILS (frontend/src/)
================================================================================

1. Routing Config & Guards (`App.jsx`):
   - Integrates `react-router-dom` for navigation.
   - Guard Components:
     * `ProtectedRoute`: Authenticated users only (redirects anonymous to `/login`).
     * `StudentRoute`: Restricts Admins from access, forcing them to `/admin`.
     * `AdminRoute`: Grants access ONLY if `user.role === 'admin'`.

2. Context Providers (`context/`):
   - `AuthContext.jsx`: Persists JWT in localStorage, provides login/logout, and 
     maintains profile states across pages.
   - `DarkModeContext.jsx`: Implements state to toggle dark mode using Tailwind.

3. API Service Client (`services/api.js`):
   - Centralized Axios client targeting backend URL.
   - Attaches `Authorization: Bearer <token>` interceptor to all requests.

4. Frontend UI Pages (`pages/`):
   - `Home.jsx`: Banner, search, recommendations, and list of upcoming fests.
   - `Events.jsx`: Complete event search engine, category and date filters.
   - `EventDetails.jsx`: Event image banners, countdown, capacity gauges.
   - `Payment.jsx`: Integrates Razorpay SDK checkout form. Provides a simulator 
     button when sandbox environment variable keys are default/missing.
   - `TicketView.jsx`: Boarding pass design with barcode QR code, printable layout 
     support, and check-in status notifications.
   - `StudentDashboard.jsx`: Leaderboards, certificate generator, registered tickets.
   - `AdminDashboard.jsx`: Event metrics, event management panels, roster downloads.
   - `AttendanceScanner.jsx`: Live camera QR feed decoder, manual input fallback.
   - `Analytics.jsx`: Recharts area/line/pie charts, hourly/daily heatmap matrix.

PART 7: SIMULATIONS & DEVELOPMENT FALLBACKS
================================================================================

To ensure development remains uninterrupted without configuration blockages:

1. Razorpay Payment Simulation:
   - Configured in `backend/routes/registrations.js`. If `RAZORPAY_KEY_ID` matches 
     `rzp_test_placeholder_key` or is missing, backend skips Razorpay order 
     creation. It creates simulated IDs and allows checkout without bank cards.
   - Frontend `Payment.jsx` renders a "Simulate Success Payment" button to bypass 
     the payment gateway, validating and generating tickets instantly.

2. QR Camera Reader Fallback:
   - Camera decoding requires HTTPS or localhost in modern web browsers. 
   - If camera access is disabled, admins can copy a ticket number (e.g. 
     `CP-HACK-SHARMA-001`) from the Event Roster and enter it into the fallback 
     input on `/scanner` to test entrance gate approvals.

3. Nodemailer Email Mocking:
   - If SMTP environment variables are missing from the backend `.env` configuration, 
     the `NotificationService` prints emails (welcome, receipts, tickets) to the 
     backend console terminal instead of failing, displaying all HTML and plain 
     text payloads.
